## Build 23 (1.0)

The app is now called **Earshot**.

### What changed
- New name throughout: the Home screen, onboarding, Settings → About, share-sheet subjects, and the Home-screen icon label all read **Earshot** instead of "Live Transcribe". The microphone-permission prompt now says "Earshot uses the microphone…".
- Nothing functional changed — same captions, same on-device transcription, same sound recognition. This is purely the rename.

### Why
"Live Transcribe" is Google's trademark (it's the name of their Android app), so it was never a safe name to ship under. "Earshot" was screened for conflicts and is clear for this space. It's also just a better name for what the app does — reading what's within earshot.

### Note
- Your saved transcripts, starred conversations, and settings all carry over — the rename doesn't touch stored data. If you had the app installed under the old name, it updates in place (same app, new label under the icon).

## Build 22 (1.0)

New captions layout option: a scrolling feed.

### Try this
- On a Live screen, tap the cog → **Layout: Focus / Feed**.
  - **Focus** (default, unchanged): the teleprompter view — one big current line, one dim previous line.
  - **Feed** (new): a chat-style scroll. Past utterances stay on screen and stack with spacing; the newest line sits at the bottom (big, with the caret) and the feed auto-scrolls up as new lines land.
- Switch between them mid-session; the choice is remembered.

### Why
Focus mode is great for glanceable 1:1, but if someone talks faster than you read, lines scroll off before you finish them. Feed mode keeps the last several turns visible so you can catch up — better for group conversations, lectures, and slower reading. It's the pattern other live-caption apps use.

We deliberately did NOT do a literal "Star Wars" perspective crawl (shrinking/receding text) — for an accessibility tool, every line stays full-size and high-contrast. The "Star Wars" idea became a clean vertical feed.

### Notes
- Both layouts respect text size, Reduce Motion (the feed jumps instead of animating when Reduce Motion is on), and VoiceOver (each line is announced; the current line carries the live-updates trait).

## Build 20 (1.0)

Stop re-downloading the Whisper model on launches when it's already cached.

### What was happening
On every launch (or every Live-screen open), the captions screen briefly showed "Downloading on-device model… 0%" even when the ~244 MB Whisper Small model was already on disk. Under the hood, `TranscriptionService.loadModel` was unconditionally calling `WhisperKit.download(...)`, which walks the model's Hub snapshot. With all files present locally that's mostly a no-op for the network but it still flashed the UI through `.loading(progress: 0)` and ran a slow file-validation pass each time.

### The fix
- Re-scan the on-disk cache on every `loadModel` call (one directory listing — cheap), so the in-memory cached set can't go stale after an init.
- If the model is already on disk → skip the `download` call entirely. Jump straight to `.compiling` and load the WhisperKit instance pointing at the local folder. The download path now only fires when the model is genuinely missing.

### What you'll see
- First launch (or after model switch): same as before — real progress bar from 0% → 100%.
- Every subsequent launch: brief "Compiling for Neural Engine" state (Core ML's prewarm step), then straight to "Listening…". No 0% download flash.

If you're STILL seeing the download progress on subsequent launches, please tell me — could be a separate path I missed, or you may be reinstalling between builds (TestFlight installs preserve the cache, but a manual delete + reinstall wipes Documents).

## Build 19 (1.0)

Putting builds back under the "Version 1.0" line in TestFlight.

Build 18 was the first build to ship with a correctly-baked marketing version — but because we'd set `MARKETING_VERSION` to `0.1.0` in project.yml (always intended as "pre-1.0 development"), Apple's TestFlight UI split it off into a brand-new "Version 0.1.0" line, separate from the 17 prior builds it had grouped under "Version 1.0" (those builds had `CFBundleShortVersionString = "1.0"` literally hardcoded in Info.plist, the source of the placeholder bug fixed in build 18).

Rather than maintain two parallel TestFlight version lines, this build bumps `MARKETING_VERSION` to `1.0` so it joins the existing line. Identical functionality to build 18 — only the marketing string differs. The in-app Settings → About → Version row now reads "1.0 (19)".

This is also the version we'll ship to the App Store, so it's the right one to standardize on.

## Build 18 (0.1.0)

Bug fix: the in-app version display.

Builds 1 through 17 all displayed "Version: 1.0 (1)" in Settings → About even though Apple's catalog tracked them at builds 6–17. The source `Info.plist` had `CFBundleShortVersionString` and `CFBundleVersion` hardcoded as "1.0" and "1" instead of the `$(MARKETING_VERSION)` / `$(CURRENT_PROJECT_VERSION)` placeholders that pick up the build settings. Build 18 is the first build where Settings → About → Version reflects the actual build number. It now reads "0.1.0 (18)".

Functionally identical to build 17 (which was identical to build 16). All the captions / sound catalog / SwiftData / accessibility / privacy work from builds 12–17 is intact.

## Build 16 (0.1.0)

Bigger sound catalog, simpler selection.

### Try this
- **Settings → Sounds → Sound detection.** Two sections now:
  - **Urgent** (top): pick up to 6 sounds that fire full-screen alerts. New "Emergency sirens" chip covers fire / police / ambulance with one slot.
  - **What we listen for** (below): ~30 non-urgent sounds, grouped by category (Pets, Home, Outdoors, Weather, Music, Human). These are auto-armed — no per-chip selection. Detections show up as the small tag in the top-right of the captions screen when "Show ambient sounds" is on.
- **Hit the cap.** Try arming 7 urgent sounds. The 7th tap is blocked and unarmed chips dim with an "alert cap reached" hint for accessibility.
- **Sirens chip.** Internally one chip → three classifier IDs (fire engine, police, ambulance). Triggering any one of those fires the alert.

### Why the change
Build 14 gave you 8 ambient chips behind their own armed set. Selecting individual ambient sounds added complexity for little benefit — the captions margin tag is meant to be passive context. Build 16 makes that path always-on (governed only by the master "Show ambient sounds" toggle) and grows the catalog from 8 to ~30 sounds across 6 categories.

### Known limitations
- The new non-urgent classifier IDs are best-guess against Apple's `version1` set. Settings → Developer → Sound classifier shows the red "Missing from this iOS" banner if any of them aren't actually present on your device. Let me know if anything's missing and I'll swap to the right ID.
- Audio classification has well-known false-positives ("wind" sometimes fires on AC noise, "music" sometimes fires on ambient hum). Margin tag auto-clears after 8 s of no fresh hit, so brief misfires fade quickly.

## Build 13 (0.1.0)

Quick follow-up to build 12: back-to-home from the captions screen.

### Try this
- Open Home → Start listening (1:1 or Group). On the captions screen, a small back chevron now sits to the left of the listening pill, top-left.

## Build 12 (0.1.0)

Captions screen, redesigned.

### Try this
- Open a Live session (1:1 or Group from Home). You'll land on the new captions screen.
- **Tap anywhere on the captions area** — floating bar slides up: cog · star · spacer · pause. It auto-hides after exactly 3 s. Tapping the bar itself does NOT reset the timer.
- **Tap the cog** → "Adjust" bottom sheet with A/A/A/A text size, "Show ambient sounds," "Vibrate on alerts," and an "All settings →" link to the full settings.
- **Tap the star** → flags the most recent finalized caption line; verify in the Summary "Starred" section.
- **Tap the pause button** → ends the session and pushes the Summary directly.
- **Stay silent for 5+ s after at least one line** → silence pill appears with breathing dot + counter ticking up.

### Visible changes since build 11

- New "Show ambient sounds" toggle (default off) — currently shows a placeholder "MUSIC AMBIENT" tag when on; real ambient detection lands in a later build.
- Group mode keeps a compact 4-speaker pill row below the status pill so attribution survives.

### Known limitations
- Real ambient-sound classification (jazz / laughter / dishes) isn't wired yet — the new toggle exists for the UI but only renders a placeholder. Urgent alerts (smoke alarm, baby crying, doorbell, CO, phone ringing) continue to work via the existing AlertView.

### Try this

- **VoiceOver.** Settings → Accessibility → VoiceOver → on. Open the Live screen — captions should announce as combined elements ("Maya: …"). New finalized lines auto-announce. Smoke-alarm alerts trap focus and announce immediately on appear.
